# 任务队列方案对比分析

## 概述

本文档对比了三种任务队列/编排方案在分布式 NSP 项目中的适用性：
- go-machinery（当前使用）
- Asynq
- MySQL（传统数据库方案）

## go-machinery（当前方案）

### 优点

1. **代码复杂度低**
   - 开箱即用，提供完整的任务编排能力（Chain、Group、Chord）
   - API 简洁，任务定义和发送代码量少
   - 抽象良好，屏蔽底层消息队列细节

2. **工作效率高**
   - 基于 Redis 的高性能消息队列，支持高并发
   - 天然支持异步任务，不阻塞主流程
   - 内置重试机制，配置简单

3. **开发效率高**
   - 文档齐全，社区活跃
   - 功能丰富，满足复杂编排需求
   - 维护成本低，框架稳定

4. **运行稳定性好**
   - 生产验证，众多公司使用
   - 支持任务状态追踪、结果存储
   - 依赖单一，仅依赖 Redis

5. **扩展性优秀**
   - 轻松增加 Worker 数量
   - 天然支持多机部署（如多 AZ 架构）

### 缺点

1. **灵活性限制**
   - 任务重试粒度粗，无法精细控制单个任务
   - 状态查询局限，缺少业务状态的深度查询
   - 自定义复杂，修改内部逻辑需要深入理解框架源码

2. **可靠性风险**
   - Redis 故障会导致任务丢失
   - 回滚复杂，需手动实现回滚逻辑
   - 任务状态与业务数据可能不一致

## Asynq

### 优点

1. **代码复杂度低**
   - API 设计现代，比 machinery 更简洁
   - 类型安全，更好的 Go 语言习惯用法
   - 轻量级，核心功能集中

2. **性能优秀**
   - 基于 Redis，性能与 machinery 相当
   - 支持灵活的重试策略和定时任务

3. **开发效率高**
   - 官方文档详细，示例丰富
   - 提供 CLI 工具和 Web UI
   - 自带 Prometheus 指标导出

4. **运行稳定性高**
   - 持久化任务到 Redis，支持任务去重
   - Web UI 实时查看任务状态
   - 支持 Worker 平滑重启

### 缺点

1. **功能限制**
   - 无内置编排，没有 Chain/Group/Chord
   - 事务支持弱，跨任务事务需要自己管理

2. **迁移成本**
   - 从 machinery 迁移需重写任务编排逻辑
   - 学习曲线虽然简单但概念不同

## MySQL（传统数据库方案）

### 优点

1. **可靠性保障**
   - ACID 事务保证，数据不丢失
   - 跨表操作强一致性，天然支持回滚
   - 所有操作可追溯，满足合规要求

2. **查询能力强**
   - 支持复杂 SQL 查询
   - 可直接生成统计报表
   - 多维度精确过滤

3. **运维成熟**
   - 备份、恢复、监控工具成熟
   - 大部分开发熟悉 MySQL
   - 海量资料和最佳实践

### 缺点

1. **性能瓶颈**
   - 轮询低效，需要定时扫描任务表
   - 并发受限，数据库连接池有限
   - 锁竞争，多 Worker 抢任务时需要行锁

2. **代码复杂度高**
   - 需要手写任务调度、状态机、重试逻辑
   - Bug 风险高，边界条件多

3. **运维成本高**
   - 轮询导致空转消耗资源
   - 需要监控任务堆积、执行时长等
   - 数据库垂直扩展成本高

## 建议

### 推荐方案：继续使用 go-machinery

基于项目特点：
1. 多 AZ 并发创建需求
2. 任务链编排需求
3. 高性能要求
4. 已有实现，切换成本高

### 优化建议

可以采用混合方案：
- 使用 machinery 做任务编排
- 使用 MySQL 存储业务状态和审计日志

### 何时考虑 Asynq

- 新项目可以从零开始选择
- 需要 Web UI 实时监控任务队列
- 简单场景不需要复杂的 Chain/Group/Chord

### 何时避免 MySQL

- 高并发场景（> 1000 TPS）
- 实时性要求高
- 团队资源有限
